第一章

STM32F407 + LAN9253：无中断简易 IO EtherCAT 从站

目标：先做一个稳定的 FreeRun IO 从站，不开启 AL Event 中断，不开启 DC，同步功能留到后续章节。

当前硬件：STM32F407 + LAN9253，4 个按键作为输入，2 个 LED 作为输出，LAN9253 使用外置 EEPROM。

目录

1. 本章目标 2. 工程结构 3. SSCTool 配置 4. PDO 与对象字典 5. 协议栈移植代码 6. IO 应用映射 7. 主循环接入 8. CMake 接入 9. EEPROM/XML 配置 10. Bug 复盘 11. 验证结果 12. 下一章计划

1. 本章目标

这一章只做最小可用从站。所谓“最小可用”，就是主站能扫描到设备，能从 Init 进入 Pre-Op、Safe-Op、Op，并能周期性读写 1 字节输入和 1 字节输出。

功能本章状态说明
CoE开启用于对象字典、PDO 映射、TwinCAT 在线查看。
RxPDO1 字节主站写给从站，控制 LED1、LED2。
TxPDO1 字节从站发给主站，表示 KEY1 到 KEY4 当前状态。
AL Event 中断关闭AL_EVENT_ENABLED = 0，协议栈轮询 AL Event。
DC 分布时钟关闭DC_SUPPORTED = 0，先不做同步。
同步模式FreeRun先保证状态机和 PDO 稳定。
通俗理解：这一章先让从站“会说话、能收发简单数据”。至于“按严格时钟节奏说话”，放到后面 DC 和中断章节。

2. 工程结构

当前工程里保留了两份 SSC 相关内容：

目录用途
SSC/SSCTool 原始生成结果和 XML/Excel 配置，方便以后重新生成。
EtherCAT/工程实际参与编译的 SSC 协议栈源码。
BSP/板级驱动，包括 SPI、LAN9253、LED、按键，以及 SSC 硬件适配层。
App/用户应用层，把 EtherCAT PDO 映射到按键和 LED。
Core/CubeMX 生成的 STM32 初始化和主函数。

整理成这种结构的好处是：以后 SSCTool 重新生成时，先覆盖 SSC/，确认无误后再同步到 EtherCAT/，自己的硬件适配和应用逻辑尽量留在 BSP/、App/ 里。

3. SSCTool 配置

本章使用的是保守配置，核心思想是“先少开功能”。

配置项值原因
SAMPLE_APPLICATION0不用官方示例应用，自己实现 APPL_InputMapping 和 APPL_OutputMapping。
USE_DEFAULT_MAIN0使用 STM32 自己的 main.c，手动调用 MainInit() 和 MainLoop()。
COE_SUPPORTED1需要对象字典和 PDO 映射。
FOE/SOE/EOE/AOE/VOE0本章不需要文件传输、网络、伺服等功能。
AL_EVENT_ENABLED0不开 LAN9253 IRQ 中断，轮询方式更容易调通。
DC_SUPPORTED0不启用分布时钟，同步功能后续再做。
ECAT_TIMER_INT0EtherCAT 定时检查放在主循环，不进定时器中断。
ESC_16BIT_ACCESS1本章保持 SSCTool 生成配置；注意它和 PDO 使用 UINT8 不冲突。
经验：不要一开始就同时打开 AL Event 中断和 DC。中断、同步、SPI 重入、ESC 事件处理会把问题混在一起。先把 FreeRun 跑稳，是比较舒服的路。

5. 协议栈移植代码

SSC 本体不认识你的 LAN9253 BSP。它只会调用一组固定名字的硬件函数，例如 HW_EscRead、HW_EscWrite、HW_GetALEventRegister。所以这里写了一个适配层，把 SSC 的调用转到现有 LAN9253 驱动。

5.2 硬件适配实现

文件：BSP/Src/bsp_ssc_hw.c

#include "bsp_ssc_hw.h"

#include "bsp_lan9253.h"
#include "stm32f4xx_hal.h"

#include <stdint.h>

static UINT32 s_timerStartMs;

UINT16 HW_Init(void)
{
    BSP_LAN9253_Init();
    s_timerStartMs = (UINT32)HAL_GetTick();

    return (BSP_LAN9253_SelfTest() != 0U) ? 0U : 1U;
}

void HW_Release(void)
{
}

UINT32 HW_GetTimer(void)
{
    return (UINT32)HAL_GetTick() - s_timerStartMs;
}

void HW_ClearTimer(void)
{
    s_timerStartMs = (UINT32)HAL_GetTick();
}

UINT16 HW_GetALEventRegister(void)
{
    return BSP_LAN9253_ReadALEvent();
}

UINT16 HW_GetALEventRegister_Isr(void)
{
    return BSP_LAN9253_ReadALEvent();
}

void HW_EscRead(MEM_ADDR *pData, UINT16 address, UINT16 len)
{
    BSP_LAN9253_EscRead((uint8_t *)pData, (uint16_t)address, (uint16_t)len);
}

void HW_EscWrite(MEM_ADDR *pData, UINT16 address, UINT16 len)
{
    BSP_LAN9253_EscWrite((const uint8_t *)pData, (uint16_t)address, (uint16_t)len);
}

void HW_EscReadIsr(MEM_ADDR *pData, UINT16 address, UINT16 len)
{
    BSP_LAN9253_EscReadIsr((uint8_t *)pData, (uint16_t)address, (uint16_t)len);
}

void HW_EscWriteIsr(MEM_ADDR *pData, UINT16 address, UINT16 len)
{
    BSP_LAN9253_EscWriteIsr((const uint8_t *)pData, (uint16_t)address, (uint16_t)len);
}

void HW_EscReadMbxMem(MEM_ADDR *pData, UINT16 address, UINT16 len)
{
    HW_EscRead(pData, address, len);
}

void HW_EscWriteMbxMem(MEM_ADDR *pData, UINT16 address, UINT16 len)
{
    HW_EscWrite(pData, address, len);
}
注意：本章虽然提供了 HW_EscReadIsr 和 HW_EscWriteIsr，但它们仍然复用阻塞式 HAL SPI。因为本章不开 AL Event 中断，所以不会在真实中断上下文频繁访问 SPI。后面开启中断时必须重新评估 SPI 重入和 ISR 内阻塞问题。

6.2 按键状态接口

原来的 Key_Poll_10ms() 是“按下事件”接口，适合菜单和触发类逻辑，但 EtherCAT 输入 PDO 更适合表达“当前状态”。所以新增了 BSP_Key_ReadStateByte()。

文件：BSP/Src/bsp_key.c

uint8_t BSP_Key_ReadStateByte(void)
{
    uint8_t state = 0U;

    if (HAL_GPIO_ReadPin(KEY_PORT, PIN_KEY1) == GPIO_PIN_RESET) {
        state |= 0x01U;
    }

    if (HAL_GPIO_ReadPin(KEY_PORT, PIN_KEY2) == GPIO_PIN_RESET) {
        state |= 0x02U;
    }

    if (HAL_GPIO_ReadPin(KEY_PORT, PIN_KEY3) == GPIO_PIN_RESET) {
        state |= 0x04U;
    }

    if (HAL_GPIO_ReadPin(KEY_PORT, PIN_KEY4) == GPIO_PIN_RESET) {
        state |= 0x08U;
    }

    return state;
}

6.3 SSC 生成文件中的应用钩子

文件：EtherCAT/Src/io_cp.c

void APPL_InputMapping(UINT16* pData)
{
    UINT8 inputByte = EcatIo_ReadInputs();

    DigitalInputs0x6000.InputByte = inputByte;

    if (pData != NULL) {
        ((UINT8 *)pData)[0] = inputByte;
    }
}

void APPL_OutputMapping(UINT16* pData)
{
    if (pData == NULL) {
        return;
    }

    DigitalOutputs0x7000.OutputByte = ((UINT8 *)pData)[0];
    EcatIo_ApplyOutputs(DigitalOutputs0x7000.OutputByte);
}

void APPL_Application(void)
{
    EcatIo_Application();
}
这三个函数位于 SSCTool 生成文件里。如果以后重新生成并覆盖 EtherCAT/Src/io_cp.c，需要把这三个钩子补回去，或者研究 SSCTool 是否能指定自定义 application file。

7. 主循环接入

之前 main.c 跑的是 LAN9253 UART 测试菜单。接入 SSC 后，主循环必须调用 MainLoop()，否则 EtherCAT 状态机和邮箱服务不会运行。

文件：Core/Src/main.c

#include "bsp_ssc_hw.h"
#include "applInterface.h"

/* ... CubeMX 初始化 ... */

MX_GPIO_Init();
MX_SPI3_Init();
MX_TIM2_Init();
MX_USART2_UART_Init();

BSP_UART_Init();
BSP_LED_Init();

if (HW_Init() != 0U) {
  Error_Handler();
}

if (MainInit() != 0U) {
  Error_Handler();
}

while (1)
{
  MainLoop();
}

通俗理解：HW_Init() 准备 LAN9253 和底层硬件，MainInit() 初始化 EtherCAT 协议栈，MainLoop() 则不断处理主站命令、状态切换、邮箱和 PDO。

9. EEPROM/XML 配置

本板有外置 EEPROM，所以本章保持：

#define ESC_EEPROM_EMULATION 0
#define ESC_EEPROM_SIZE 0x800

XML 里 EEPROM 大小为 2048 字节，并填入 ESC 配置数据：

<Eeprom>
  <ByteSize>2048</ByteSize>
  <ConfigData>800E00CC8813FF00000040C00000</ConfigData>
</Eeprom>
改 XML 不等于已经把外置 EEPROM 烧好。XML 是给 TwinCAT/主站看的 ESI 描述，真正上电启动还依赖 LAN9253 外置 EEPROM 中实际保存的 EEPROM 内容。

10. Bug 复盘

10.1 缺少 SSC 硬件适配层，编译失败

现象：加入 EtherCAT/Src 后编译报大量 implicit declaration of function 'HW_EscReadWord'、HW_GetALEventRegister、HW_EscReadMbxMem 等错误。

原因：SSC 协议栈是硬件无关代码，它假设平台移植层已经提供 HW_* 函数/宏。LAN9253 BSP 虽然已经能读写 ESC，但名字和 SSC 要求不同。

修复：新增 BSP/Inc/bsp_ssc_hw.h 和 BSP/Src/bsp_ssc_hw.c，把 SSC 的 HW_* 调用转接到 BSP_LAN9253_EscRead/EscWrite。

10.2 APPL_InputMapping / APPL_OutputMapping 为空

现象：SSCTool 生成代码能编译一部分，但 io_cp.c 中有 #warning "Implement input"、#warning "Implement output"。

原因：SAMPLE_APPLICATION = 0 表示不用官方示例应用，所以用户必须自己实现应用映射。

修复：在 App/ 中实现 IO 应用逻辑，并在 io_cp.c 三个应用钩子里调用。

10.3 TwinCAT 进不了 SafeOP：0x001e Invalid SM IN cfg

现象：TwinCAT 状态停在 ERR PREOP，错误为：

PREOP to SAFEOP failed
AL Status Code 0x001e - Invalid SM IN cfg

直觉误区：一开始容易怀疑 EEPROM、SM3 地址、PDO 长度或 TwinCAT 配置。但 XML 中 SM3 是 StartAddress=0x1400、DefaultSize=1、ControlByte=0x20，对象字典中 TxPDO 也是 1 字节。

真正原因：GCC 默认会给结构体成员补齐。例如 UINT16 u16SubIndex0; UINT32 SI1; 中间会补 2 字节。如果 SSC 的对象字典偏移计算按紧凑布局读取，就会读偏 0x1A00 PDO 映射项，导致协议栈算错 nPdInputSize。主站写的 SM3 长度明明是 1，协议栈内部期望值却不是 1，于是报 Invalid SM IN cfg。

修复：在 bsp_ssc_hw.h 里定义 STRUCT_PACKED_END、MBX_STRUCT_PACKED_END、OBJ_STRUCT_PACKED_END 为 __attribute__((packed))，并设置 OBJ_WORD_ALIGN=0、OBJ_DWORD_ALIGN=0。

结果：重新编译烧录后，设备可以进入 SafeOP/OP，IO 功能正常。

10.4 关于 AL_EVENT_ENABLED 和 DC

本章不建议打开 AL_EVENT_ENABLED 和 DC_SUPPORTED。原因是当前 HW_EscReadIsr/HW_EscWriteIsr 仍然复用阻塞 HAL SPI。如果真的在 EXTI 中断里直接访问 SPI，可能引入阻塞、重入和时序问题。

后续要打开 AL Event 中断时，建议先设计“中断只置位标志，主循环处理 SPI”的方式，或者系统性评估 SPI/DMA/临界区。

11. 验证结果

项目结果
编译通过，生成 build/Debug/EtherCAT_Slave_STD.elf。
TwinCAT 扫描能识别 IO 从站。
状态机可以进入 SafeOP 和 OP。
输入按键状态通过 Input Byte 上传。
输出主站写 Output Byte 可控制 LED1/LED2。
本章结论：无中断、无 DC 的简易 IO 从站已经打通。现在这个版本适合作为后续章节的稳定基线。

12. 下一章计划

建议后续按这个顺序继续，不要一次打开太多功能：

  1. 整理 TwinCAT 侧 ESI 导入、EEPROM 烧录和重新扫描流程。
  2. 加入更清晰的调试打印，例如状态机变化、AL Status Code、SM 配置读取。
  3. 打开 AL_EVENT_ENABLED，但先不要开 DC，验证 LAN9253 IRQ 事件处理。
  4. 稳定后再开启 DC，验证 Sync0/Sync1、周期时间和 0x1C32/0x1C33。
  5. 最后再考虑输入消抖策略是否放入 PDO 状态，或者保留原始 GPIO 状态。

复习时记住本章主线：SSCTool 生成协议栈和对象字典，BSP 提供 ESC 读写，App 负责 PDO 与硬件的含义映射，main.c 周期调用 MainLoop。
